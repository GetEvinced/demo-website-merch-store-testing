1. Audit Overview

This report presents the results of a full-site automated accessibility audit of the Demo Website (a React 18 SPA served on http://localhost:3000). The audit was performed using the Evinced accessibility engine (MCP Server: EVINCED_MCP_SERVER_JFROG, package @evinced/mcp-server-web@1.0.28), which applies four complementary detection strategies:

  • AXE — Standard axe-core rules (WCAG A/AA violations)
  • GEN1 — Evinced first-generation engine: interactable role & keyboard accessibility
  • GEN2 — Evinced component-pattern engine: navigation, combobox, checkbox, dialog patterns
  • GEN3 — Evinced contextual engine: reflow, reading order, heading structure, focus order

A full user journey was traced: Homepage → Products Listing → Product Detail → Cart Drawer → Checkout → Order Confirmation. Additionally, targeted component scans were run on the sort combobox widget and main navigation menu on the Products page. No remediation was performed — this report documents findings only.

4. Critical Issues — Detailed Findings & Recommended Fixes

The following 53 critical issues represent WCAG Level A/AA violations that prevent or severely impair access by screen reader users and keyboard-only users. No remediations were applied to the codebase; recommended fixes are provided for each.

A. Missing Image Alternative Text (AXE: image-alt) — 2 Issues

A-1: Hero banner image missing alt attribute

AXE image-alt  ·  WCAG 1.1.1 (A) — Non-text Content  ·  Critical
PageHomepage
Filesrc/components/HeroBanner.jsx
Element<img src="/images/home/New_Tees.png">
DescriptionThe hero banner image has no alt attribute. Screen readers will fall back to announcing the raw filename ("New underscore Tees dot png"), which conveys no meaningful information to a blind user.

Recommended Fix

Add a descriptive alt attribute that conveys the image content and marketing intent:

<img src="/images/home/New_Tees.png" alt="New winter basics collection — seasonal tees and essentials" />

If the image is purely decorative and the adjacent headline already conveys the meaning, use an empty alt instead: alt="".

Why This Approach

WCAG 1.1.1 requires all non-text content to have a text alternative. A descriptive alt is the lowest-friction, universally-supported solution — it works across all assistive technologies with no JavaScript dependency. Empty alt="" is used only when the image is purely decorative and nearby text provides the same information.

B. Invalid ARIA Attribute Values (AXE: aria-valid-attr-value) — 3 Issues

B-1: aria-expanded="yes" on a heading element

AXE aria-valid-attr-value  ·  WCAG 4.1.2 (A)  ·  Critical
PageHomepage
Filesrc/components/FeaturedPair.jsx
Element<h1 aria-expanded="yes">
Descriptionaria-expanded accepts only "true" or "false". The value "yes" is not a valid ARIA token; assistive technologies will ignore it entirely, leaving the state of the widget unannounced.

Recommended Fix

Replace the invalid value and place aria-expanded only on interactive elements (buttons/links), not headings:

<!-- Remove aria-expanded from the heading entirely (headings are not interactive) -->
<h2>Featured Items</h2>

<!-- If expand/collapse behaviour is needed, put it on the trigger button: -->
<button aria-expanded={isExpanded} aria-controls="featured-panel">
  Featured Items
</button>

Why This Approach

aria-expanded communicates widget state; it belongs on the interactive trigger (button/link), not the heading. Placing it on a heading creates a false contract with assistive technologies. Correcting both the value and the element type ensures ATs can reliably announce expanded/collapsed state.

B-2: aria-sort="newest" — invalid token on columnheader

AXE aria-valid-attr-value  ·  WCAG 4.1.2 (A)  ·  Critical
PageProducts Page
Filesrc/pages/NewPage.jsx
Element<div aria-sort="newest" role="columnheader">
Descriptionaria-sort accepts only ascending, descending, none, or other. The custom value "newest" is invalid and will be ignored by assistive technologies.

Recommended Fix

<!-- Use the closest valid token. For "newest first" (descending by date): -->
<div role="columnheader" aria-sort="descending">Newest</div>

Why This Approach

ARIA specification requires enumerated tokens. Replacing "newest" with "descending" faithfully maps the intent (most-recent items first) to the closest valid ARIA concept, allowing screen readers to announce the current sort direction correctly.

B-3: aria-relevant="changes" — invalid token

AXE aria-valid-attr-value  ·  WCAG 4.1.2 (A)  ·  Critical
PageProduct Detail
Filesrc/pages/ProductPage.jsx
Element<ul aria-relevant="changes">
Descriptionaria-relevant accepts space-separated tokens from: additions, removals, text, all. The value "changes" is not valid.

Recommended Fix

<!-- Use a valid token combination (e.g. additions + text for a live list): -->
<ul aria-relevant="additions text" aria-live="polite">

Why This Approach

The intent of aria-relevant="changes" is to announce both added/removed content. Using "additions text" is the standardised approximation for "announce when content is added or changes".

C. Duplicate ARIA IDs (AXE: duplicate-id-aria) — 4 Issues

C-1 & C-2: Duplicate ARIA IDs in FeaturedPair component

AXE duplicate-id-aria  ·  WCAG 4.1.1 (A)  ·  Critical
PageHomepage
Filesrc/components/FeaturedPair.jsx
ElementTwo <p id="featured-card-label"> elements and two <img id="featured-card-img"> elements (rendered in a .map() loop)
DescriptionWhen the same id appears multiple times, an ARIA aria-labelledby or aria-describedby referencing that ID resolves to the first matching element only. The second card's label/image relationship is broken for assistive technologies.

Recommended Fix

Make IDs unique by incorporating the loop index or item identifier:

{featuredItems.map((item, index) => (
  <div key={item.id} className="featured-card">
    <img id={`featured-card-img-${item.id}`} src={item.image} alt={item.imageAlt} />
    <p id={`featured-card-label-${item.id}`} className="featured-eyebrow">
      {item.eyebrow}
    </p>
  </div>
))}

Why This Approach

HTML IDs must be unique within a document. Namespacing with a stable item identifier (not just the loop index, to survive reordering) guarantees uniqueness while preserving the ARIA label relationships. Using item.id is preferred over index because it remains stable if the array is sorted or filtered.

C-3 & C-4: Duplicate filter-section-title ID in FilterSidebar

AXE duplicate-id-aria  ·  WCAG 4.1.1 (A)  ·  Critical
PageProducts Page
Filesrc/components/FilterSidebar.jsx
ElementThree <span id="filter-section-title"> elements for Price, Size, and Brand sections
DescriptionEach filter group's header button references this ID via aria-describedby. The repeated ID causes all three buttons to use the same description (the first one in DOM order), corrupting the ARIA labelling of the Price and Size disclosure buttons.

Recommended Fix

  <div key={group.name} className="filter-group">

))}

Why This Approach

Same principle as C-1/C-2. Embedding the group name (e.g., filter-title-price) in the ID ensures each section title is uniquely referenceable, restoring the correct ARIA description for each disclosure button.

G. Sort Combobox / Listbox Failures (GEN2) — 4 Critical Issues

G-1 through G-4: Sort combobox and listbox missing roles, names, and keyboard support

GEN2 sort-combobox · sort-dropdown  ·  WCAG 4.1.2, 2.1.1 (A)  ·  Critical
PageProducts Page
Filesrc/pages/NewPage.jsx
#RuleElementIssue
G-1sort-combobox-no-role<button class="sort-btn">role="combobox" not set — AT cannot identify this as a combobox trigger
G-2sort-combobox-no-accessible-name<button class="sort-btn">aria-label removed — no accessible name
G-3sort-dropdown-no-listbox-role<ul class="sort-options">role="listbox" and aria-label removed — not identified as a listbox
G-4sort-dropdown-no-option-role + no-keyboard-accessible<li class="sort-option">role="option", aria-selected, tabIndex, and keyboard handler removed — options are invisible and unreachable via keyboard

Recommended Fix — Restore full combobox ARIA pattern

<div role="group" aria-label="Sort products">
  <button
    className="sort-btn"
    role="combobox"
    aria-label="Sort by"
    aria-expanded={sortOpen}
    aria-haspopup="listbox"
    aria-controls="sort-listbox"
    onClick={() => setSortOpen(!sortOpen)}
  >
    {selectedSort}
  </button>

  {sortOpen && (
    <ul id="sort-listbox" role="listbox" aria-label="Sort options">
      {sortOptions.map((option) => (
        <li
          key={option.value}
          role="option"
          aria-selected={selectedSort === option.label}
          tabIndex={0}
          onClick={() => handleSort(option)}
          onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') handleSort(option); }}
        >
          {option.label}
        </li>
      ))}
    </ul>
  )}
</div>

Why This Approach

The ARIA listbox/combobox pattern is the standard way to implement custom selects. Using role="combobox" on the trigger and role="listbox" + role="option" on the list establishes the semantic relationship between the trigger and the options panel. aria-expanded communicates open/closed state; aria-controls links the trigger to the options panel. Each option must be keyboard reachable and have aria-selected to convey selection state. Alternatively, replacing the custom widget with a native <select> element would eliminate all of these requirements at once.

role="dialog" + aria-modal="true" is the WAI-ARIA standard for modal overlays. It signals to screen readers that they should restrict virtual cursor navigation to the modal's contents. aria-label on the dialog identifies its purpose. Contextual labels on quantity buttons (including the item name) allow users acting out-of-context (e.g., review screen content) to know which item is being modified. Focus management (moving focus into the modal and trapping it there) is required by WCAG 2.1.2 so keyboard users cannot accidentally interact with obscured background content.

J-2: Checkout form validation errors are silent — role="alert" removed from error spans

UNDETECTABLE live-region  ·  WCAG 4.1.3 (AA) Status Messages  ·  Critical
PageCheckout
Filesrc/pages/CheckoutPage.jsx
Elements#firstName-error, #lastName-error, #address-error, #cardNumber-error, #expirationDate-error
DescriptionWhen form validation fails, up to 5 error messages are injected into the DOM. Previously each carried role="alert" (assertive live region), causing immediate screen reader announcement. Without it, errors appear visually but are completely silent — blind users cannot know the form submission failed.

Recommended Fix — Restore role="alert" on error spans + link errors to fields

<!-- Restore role="alert" and aria-describedby association -->
<input
  id="firstName"
  type="text"
  aria-describedby="firstName-error"
  aria-invalid={!!errors.firstName}
/>
{errors.firstName && (
  <span id="firstName-error" role="alert" className="form-error">
    {errors.firstName}
  </span>
)}

Why This Approach

role="alert" creates an assertive live region that announces its content immediately when injected into the DOM — exactly the right behaviour for critical form errors. Pairing it with aria-invalid="true" on the field and aria-describedby pointing to the error span ensures the error is also announced when the user re-focuses the invalid field, providing two channels of discovery for keyboard users.
